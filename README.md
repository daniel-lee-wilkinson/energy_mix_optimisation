# Energy Mix Optimisation Project

This project performs energy mix optimisation analysis for renewable energy sources, specifically focusing on minimising the Global Warming Potential (GWP) per kWh supplied.

## File Naming Convention

Output files (figures and data) generated by `run_optimization.py` are saved with fixed names in their respective directories (`figures/`, `output_data/`). Each run of the script overwrites the files from the previous run.

## Setup

1. Create a Python virtual environment (recommended).
2. Install the dependencies: `pip install -r requirements.txt` (ensure pandas, numpy, matplotlib, scipy, requests, pytz are included)
3. Ensure the input weather data file (`output_data/australian_energy_data_potentials.csv`) is present. If not, you may need to run `run_optimization.py` once with default settings to generate it.
4. Optionally set environment variables for location coordinates (see Environment Configuration section). If unset, built-in default coordinates are used.
5. Run the main hourly optimisation script:
   ```bash
   python run_optimization.py
   ```
   You will be prompted to enter the target annual energy demand in MWh.

6. (Optional) Run the simplified Linear Programming optimisation script:
   ```bash
   python run_lp_optimization.py
   ```
   You will be prompted to enter the target annual energy demand in MWh.

7. (Optional) Run demand sensitivity analysis to visualise PV/Wind/Grid share changes with demand:
   ```bash
   python run_demand_sensitivity.py
   ```
   This generates:
   - `output_data/demand_sensitivity_results.csv`
   - `figures/demand_sensitivity_energy_mix.png`
   - `figures/demand_sensitivity_capacities.png`

## Environment Configuration

The script reads environment variables (via `python-dotenv`) for location configuration:
* `PRIMARY_LOCATION_NAME`: Name of the primary site
* `PRIMARY_LATITUDE`: Latitude of the primary site
* `PRIMARY_LONGITUDE`: Longitude of the primary site
* `FALLBACK_LOCATION_NAME`: Name of the fallback site
* `FALLBACK_LATITUDE`: Latitude of the fallback site
* `FALLBACK_LONGITUDE`: Longitude of the fallback site

Optional analysis period overrides (`YYYY-MM-DD`):
* `ANALYSIS_START_DATE`: Start date for full runs (default `2023-01-01`)
* `ANALYSIS_END_DATE`: End date for full runs (default `2023-12-31`)
* `QUICK_TEST_START_DATE`: Start date for quick test mode (default `2023-01-01`)
* `QUICK_TEST_END_DATE`: End date for quick test mode (default `2023-01-07`)

Optional capacity search overrides:
* `PV_CAPACITY_MIN_MW`: Lower bound for PV capacity search range
* `PV_CAPACITY_MAX_MW`: Upper bound for PV capacity search range
* `WIND_CAPACITY_MIN_MW`: Lower bound for Wind capacity search range
* `WIND_CAPACITY_MAX_MW`: Upper bound for Wind capacity search range
* `CAPACITY_STEPS`: Number of points in each capacity search grid (must be >= 2)
* `ANNUAL_CO2_REMOVAL_TONNES`: Annual atmospheric CO2 removal credit used for net GWP reporting (default `5000`)

You can set these variables in either:
- your shell environment, or
- a `.env` file in the project root (auto-loaded if present).

If no environment variables are provided, the script uses anonymous non-site-specific defaults.
For public releases, set explicit coordinates in your local `.env` and keep `.env` out of version control.

## Overview & Example Scenario

The primary script (`run_optimization.py`) simulates an energy system hourly over a year (using NASA POWER data for a remote site in central Australia) to find the mix of PV and Wind capacity that minimizes the overall GWP per kWh supplied. It considers:

*   A target annual energy demand (user-defined).
*   A land area constraint (e.g., 1000 km²).
*   A fixed battery energy storage system (120 MWh capacity, 90 MW power, 90% efficiency, 20% min SoC).
*   Specified GWP factors (kg CO2e/kWh): PV=0.07, Wind=0.011, Grid=0.6.

**Example Use Case:** The system was used to determine the lowest-GWP energy supply mix required to meet a 10,000 MWh annual demand for a hypothetical CO2 removal system operating at a remote central Australian site within a 1000 km² land area. The results indicate that maximizing PV capacity (50 GW) within the land limit provides the lowest GWP energy, despite requiring significant grid backup (~50% of energy).

## Features

- Fetches real weather data from NASA POWER API
- **Primary Goal:** Calculates optimal energy mix (PV/Wind) to minimise global warming potential (GWP) per kWh.
- Incorporates a fixed-size battery energy storage system (BESS) model.
- Analyses temperature effects on PV efficiency and energy demand.
- Respects user-defined annual energy demand and land area constraints.
- Creates visualisations of energy generation profiles and optimisation results.

## Data Sources

*   **Primary:** Real-time hourly weather data (Solar Irradiation, Wind Speed, Temperature) is sourced from the [NASA POWER Project](https://power.larc.nasa.gov/) API for a remote site in central Australia. Coordinates can be configured via environment variables; if not set, built-in defaults are used.
*   **Fallback:** If the primary site data is unavailable, the system attempts to use data from a secondary remote Australian site or generates synthetic data.

## Assumptions

The hourly simulation model (`run_optimization.py`) relies on the following key assumptions:

**General:**
*   **Location:** A remote site in central Australia. Coordinates are loaded via environment variables.
*   **Time Resolution:** Hourly.
*   **Data Source:** Prefers real weather data from NASA POWER API for the specified location.

**PV Generation:**
*   **Efficiency:** Assumed constant at 15%.
*   **Temperature Effect:** Performance degrades by 0.4% for every 1°C increase above 25°C.
*   **Land Use:** Requires 0.02 km² of land per MW of installed capacity.
*   **GWP (Lifecycle):** 0.07 kg CO2e per kWh generated.

**Wind Generation:**
*   **Power Curve:** Standard curve (Cut-in: 3 m/s, Rated: 12 m/s, Cut-out: 25 m/s).
*   **Land Use:** Requires 0.26 km² of land per MW of installed capacity.
*   **GWP (Lifecycle):** 0.011 kg CO2e per kWh generated.

**Battery Storage:**
*   **Capacity:** Fixed at 120 MWh.
*   **Power:** Fixed at 90 MW (charge/discharge).
*   **Efficiency:** 90% round-trip (applied on discharge).
*   **Min SoC:** Fixed at 20%.
*   **Dispatch:** Charges only from excess PV/Wind; Discharges to meet demand *before* grid import.

**Grid Supply:**
*   **Availability:** Assumed infinitely available to meet any energy deficit.
*   **GWP:** 0.6 kg CO2e per kWh.

**Energy Demand:**
*   **Annual Target:** User-specified `annual_demand_mwh`.
*   **Temperature Adjustment:** Base hourly demand increased by 20% when temperature > 30°C.
*   **Scaling:** Profile scaled to match the annual target exactly.

**Optimisation (`optimise_land_use` function):**
*   **Objective:** Minimise the average GWP per kWh of the energy supplied (PV + Wind + Grid).
*   **Constraints:** Total land use <= available land (e.g., 1000 km²); energy supplied must meet demand.
*   **Method:** Grid search over dynamically calculated PV and Wind capacity ranges.

## Simplified LP Model (`run_lp_optimization.py`)

An alternative script, `run_lp_optimization.py`, uses Scipy's linear programming solver. **Note:** This model makes significant simplifications:
*   Uses *annual average* generation potentials, not hourly data.
*   **Does not model the battery.**
*   Optimizes total annual energy contributions (kWh) from PV, Wind, and Grid.
It provides a different, high-level perspective based purely on annual averages and land constraints.
